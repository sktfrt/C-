# Исключения

> try-catch не ловит UB

Обработка исключений - способ обрабатывать ошибки

В Си приходилось использовать глобальные переменные, переменные контекста и т.п. Неудобно для обработки маллока например.

В С++ появились исключения, которые помогают спокойно закрыть ресурсы при аварийном завершении программы

std::exception

Популярные исключения:

- std::logic_error
- std::runtime_error
- std::out_of_range
bad_optional_access

Сами по себе названия исключений говорят не так много. Иногда хочется что-то более информативное, чем logic_error и runtime_error

## Бросание исключений

Некоторые варианты поймать исключение:
1. new - может бросать исключения
1. Методы std - std::vector.at(...)
1. throw exception;

`throw;`, само по себе - отдельный разговор

Если пишем функцию, которая не бросает исключения, можно в конце написать `noexception` - говорит компилятору, что она не бросает исключение

Если она все же бросит исключение, то программа сразу завершится через terminate - не оправдали доверие компилятора

## Появление исключения

Вызваются деструкторы у всех объектов на пути, пока не будет кем-то отловлено - раскрутка стека

Если оказывается в try-catch то пробегается по блокам catch и ищет себя

Когда обработка прошла, исполнение продолжается с того момента, где обработано

## throw in catch

Можно бросить ошибку внутри catch

Если выделяем память в конструкторе, возникает ошибка, в catch все закрываем, но наружу все еще хотим вернуть наверх ошибку "объект не создан, состояние не валидное"

Можно создать флаг bool isValid и всегда обращаться к нему

Вариант лучше - бросить ошибку дальше. Наш блок catch закрыл ресурсы, мы бросаем ошибку и уже сам факт возникновения ошибки отлавливается сверху

Для этого и используется слово `throw;` без аргументов. Попытка написать так вне кэтч - ошибка компиляции

> `throw;` сохраняет всю информацию о исключении, если мы бросаем новое исключение, то его контекст будет уже другой.

## Раскрутка стека

Когда исключение бежит наверх - вызываются деструкторы у всех встреченных объектов

Если же возникнет исключение в деструкторе, то программа сразу же завершится терминально

> Деструкторы не должны бросать исключение

### Деструкторы

Хотя не хотим бросать исключения наружу, блок try-catch внутри деструктора абсолютно нормально

> Когда говорят, что функция не бросает исключений, говорят именно о их бросании **наружу** - если обаботка покрывает все исключения, то ок

## Собственное исключение 

По идее, try охватывает собой все тело конструктора, (естественно включая список инициализации).

Через throw можно бросить любой объект

    throw 2;
    throw "error"

Поймать можно через 

    catch(int x) {}
    catch(std::string s) {}
    catch(...) {}

Часто везде бросатся std::runtime_error, std::logic_error

Но вместо этого можно написать свой класс исключения, которое наследуется от std::exception

### Наследник std::exception

```cpp
class std::exception
{
public:
    virtual std::string what() = 0;
};
```

```cpp
class my_exception : public std::exception // Название, естественно, должно быть говорящим
{
public:
    my_exception(std::string msg) : message_(msg) {}

    std::string what() override
    {

    }

private:
    std::string message_;
};
```

```cpp
int main()
{
    try {
        throw my_exception("bad"); // Сообщение, естественно, должно быть информативным
    }
    catch (const my_exception &e)
    {
        cstd::cerr << e.what() << std::endl;
    }
    return 0;
}
```

В целом, можно использовать runtime_error и logical_error:

- runtime_error - ошибки выделения памяти, нарушения инвариантов
- logical_error - некорректные данные, полученные извне и т.п.

## Идиома RAII

В больших проектах new нужно вызывать **по минимуму**. Перед тем как написать new, нужно подумать, можно ли обойтись без него.

Идея исключения: логика написана где-то как, а обработка в соверешенно другом куске, находящемся далеко от логики

Но если мы пишем new, то сразу же по-хорошему должны обернуть его try-catch, из-за чего как раз происходит смешение

Есть другие способы выделения динамической памяти

Обертки над new из std - vector, shared_ptr, map и т.п. 

Они включают выделение памяти, не нужно думать об обработке catch

Хотим, чтобы вся очистка динамической памяти происходила в деструкторах

---

У вектора
- `[]` - бросает UB в случае некорректного доступа
- `.at()` - бросает исключение в случае некорректного доступа

`[]` работает быстрее. Если в глубинах программы какой-то индекс уже проверен много раз, то лучше использовать напрямую `[]`.

## Гарантии исключений

### Без гарантии

Если помираете, к вам никто не подойдет. Вы не гарантируете ничего и не берете никакой отвественности. Возмонжость возникновения terminate, падение программы, нарушение инвариантов.

### Базовая гарантия

*Считается, что у не плохой программы должна быть хотя бы базовая гарантия*

Инварианты программы сохраняются всегда.

Также вызывабися деструкторы, очищается память.

После обработки и выбора исключения состояние программы адекватное - клиент может считать, что инварианты исползуемого класса сохранались

### Сильная гарантия / Строгая гарантия

Если исключение бросается, то состояние программы откатывается до момента, когда все было хорошо (до вызова плохой функции).

### Не бросает исключений

Говорит за себя. Если компилятор встречает исключение, то происходит terminate из-за нарушения гарантии.

Обеспечивается директивой `noexcept`.

*Вообще говоря, возникновение **UB** никак не связано с уровнем гарантий исключения. Но в операторе `[]` у std::vector нет гарантии `noexcept`, хотя не бросает исключений.*

# inline, static

## inline

Сообщает компилятору: у этой функции может быть много реализаций, но они все одинаковы. Избавление от Multiply definition

Следствие написания inline - ускорение работы, путем замены вызов функцией подстановкой кода

**Начиная с C++17**:

    // header.hpp
    inline const std::string x = "Hello";

Без `inline` к глобальным констатам в хедере автоматически приписывается слово `static` - создай экземляр для каждого .cpp файла, которые подключает хедер (по умолчанию `inline` не приписывается в хедере).

Через него нельзя пробрасывать значения, потому что своя для каждого файла

$\Rightarrow$ если поставить большой объект, например map или vector, то может быть большая наргузка на память

`inline` позволяет создать только одну такую константу для всех .cpp

У методов классов автоматически ставится inline - поэтому можно написать релазиацию и в хедере, и в .cpp

---

`inline constexpr` - часто лучше, чем `inline const`. О нем будем говорить позже, но можно почитать о нем в документации.

Это выражение константы на этапе компиляции.

# Singleton

Ход мыслей для построения паттерна:

```cpp
// Logger.hpp
class Logger {
private:
    static bool was_created = false; // Чтобы сосздать только один экземпляр

    Logger() = default; // Хотим, чтобы объект был единственным
    Logger(const Logger &other) = default; // Тогда надо запретить и копирование
public:
    static Logger *create(); // Хотим получать объект без небходимости создать его самостоятельно
};
```
```cpp
// Logger.cpp
#include "Logger.hpp"

Logger *create()
{
    if (was_created == false) // Создаем только один раз
    {
        was_created = true;
        return new Logger();
    }
    
    else
        return nullptr;
}
```
