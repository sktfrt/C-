# Pointers

`unique_ptr` -- единственный указатель на память (но его надо перемещать и передавать через move)

`shared_ptr` - пока будем юзать его

`weak_ptr` - способ избежать проблемы, которая возникает при shared_ptr



```cpp

class Cat
{
    public:
        shared_ptr<Cat> friend_cat = nullptr;
        string name;
};

int main()
{
    shared_ptr<Cat> cat1 = make_shared();
    shared_ptr<Cat> cat2 = make_shared();

    cat1->friend_cat = cat2; //теперь на cat2 указывает два shr_ptr => счетчик = 2
    cat2->friend_cat = cat1; // на cat1 счетчик = 2

    // дестркутор cat1 => counter-- => cat1.counter = 1
    // Аналогично cat2.counter = 1
    // на момент деструктора cat2, cat1 все еще жив т.к. есть указатель изнутри cat2
    // cat2 не исчезнет, так как есть указатель изнутри cat1
    // cat1 не исчезнет, так как есть указатель изнутри cat2

    //Оба останутся
}
```

Поэтому хотим использовать weak_ptr`

```cpp

class Cat
{
    public:
        weak_ptr<Cat> friend_cat = nullptr;
        string name;
};

int main()
{
    shared_ptr<Cat> cat1 = make_shared();
    shared_ptr<Cat> cat2 = make_shared();

    cat1->friend_cat = cat2;
    cat2->friend_cat = cat1;
}
```

`weak_ptr` не увеличивает счетчик, когда смотрит на кого-либо

Можно заблокировать с помощью 

```cpp
shared_ptr p1 = ...;
weak_ptr ptr(p1);
ptr.lock(p1);
```

`weak_ptr` лучше не разбрасываться

# Списки инициализации

```cpp

Matrix(int r, int c) : rows(r), cols(c) {} // Список инициализации

Matrix(int r, int c) /* Здесь происходит инициализация всех полей */ { 
    // Здесь все остальное
    rows = r; 
    cols = c;
}
```

При использовании списка инициализации поля не инициализируются мусором до фигурных скобок, а сразу заполняются чем-то конкретным

> `Matrix {}` -> поля заполняются не мусором

Если написать чему равно поле, то укажем ему значение по умолчанию - можно написать в классе

## Порядок заполнения полей

1. **Значение по умолчанию** type variable = value;
2. **Список инициализации**  : varible(value) {} // - приоритетнее default значения
3. **Тело конструктора**  

Поля инициализируются в том же порядке, в котором объявлены в классе

`-Worder` чтобы компилятор указывал на беспорядок в списке инициализации

> В списке инициализации можно вызвать конструктор member-класса

По факту основная задача списка инициализации

```cpp
Matrix(int x, int b, int a) : varibale(x, a) {}
```

## Делегирование конструкторов

```cpp
class Matrix
{
public:
    Matrix(int x);

    Matrix(int a, int b);

}

// cpp:

Matrix::Matrix(int a, int b) {}

Matrix::Matrix(int x) : Matrix(x, x) {}
```

**Списки инициализации недоступны при делегирования конструктора!**

Но в теле конструктора можно перезаписать тем, что надо - не будет только подконтрольной инициализации

**При делегировании конструкторов нет никакой защиты от циклов!!**

# Наследование

## final

`final` - говорит что у класса не может быть наследников / метод нельзя переопределить в производном классе

```cpp
class Base final // обозначаем класс final - никто не сможет наследовать от него
{

}
```

## virtual // override

`virtual` нужно ставить только в родительском, во всех дочерних проставляется автоматически.

`override` указывает компилятору, что данный метод **должен кого-то переопределить**.

## Конструкторы деструкторы

Конструкторы и деструкторы не наследуются

```cpp
class Base
{

}

class Child : public Base
{
    Child() // Вызовется коснтруктор Base по умолчанию
    {

    }
}
```
```cpp
class Base
{
    Base(int x)
}

class Child : public Base, public Base2
{
    Child(int x) : Base(10) /* вызывается в самом начале */, Base2(x), name("")  
    {

    }
}
```

- Деструктор всегда создаётся по умолчанию.  

- Всегда явно указываем деструктор в случае наследования

- Деструктор потомка всегда вызывает деструктор родителя

- Если не написать virtual у деструктора, то при delete указателя на родителя, вызовется деструктор родителя даже тогда, когда по указателю лежит наследник

**Всегда пишем virtual у базовых классов**

> Виртуальные функции не надо вызывать в деструкторе, потому что это ломает логику

Потому что если базовый юзает виртуальный метод, то после подчищения производного класса базовый вызовет опять же вирутальный метод, принадлежащий наследнику

*Но все ещё можно вызывать метод класса статически `Base::print();`*

---

Для дефолтных конструкторов производных классов можно использовать

using Base::Base

Чтобы автоматически использовать конструктор базового класса
